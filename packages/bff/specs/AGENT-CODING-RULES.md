## 1. Absolute Prohibitions

These are hard stops. There are no exceptions. No exceptions for "just this once,"
"it's temporary," or "the type is too complex."

### 1.1 Suppression directives are forbidden

```
// @ts-ignore          ← FORBIDDEN
// @ts-expect-error    ← FORBIDDEN
// eslint-disable      ← FORBIDDEN (any form, any rule, inline or block)
// eslint-disable-next-line  ← FORBIDDEN
/* eslint-disable */   ← FORBIDDEN
```

If TypeScript or ESLint is complaining, **fix the underlying problem**.
A suppression comment is evidence that something is wrong with the code, not the linter.

**What to do instead:**

- Type errors → fix the types, add a proper type guard, or use a typed assertion (`as ConcreteType`, only when you have proven the type is correct)
- `any` type errors → define a proper interface or use `unknown` with a type guard
- Unused variable → remove it or prefix with `_` if it's a required destructure parameter
- ESLint rule complaint → comply with the rule

### 1.2 `any` is forbidden except at exact boundaries

`any` is only permitted in two places:

1. Inside a `catch` block parameter — use `unknown` and narrow it: `if (error instanceof Error)`
2. When consuming a third-party library that itself types something as `any` — wrap it immediately in a typed function that returns a concrete type

Everywhere else: `unknown`, generics, or concrete types.

### 1.3 Non-null assertion operator requires a comment

```typescript
// BAD
const user = result!;

// REQUIRED form — only when you have proven it cannot be null
// SAFETY: user is guaranteed non-null here because we checked campaign.creator_id
// in the guard above and the DB join always returns the creator row.
const user = result!;
```

If you cannot write that comment truthfully, do not use `!`.

### 1.4 No new packages

Do not run `pnpm add`, `pnpm install <package>`, or modify any `package.json`
`dependencies` or `devDependencies` to add a package that isn't already present.

**The only exception:** a package explicitly named in the spec's technology stack
(section 0 of system-spec-v2.md) that is not yet installed. In that case, install
it in the correct package only, note it in a comment at the top of the file that
uses it, and stop — do not install it transitively elsewhere.

Do not install alternative packages. If the spec says `pdf-parse`, do not install
`pdfjs-dist`. If the spec says `bull`, do not install `bullmq`.

### 1.5 No new workspace packages

Do not create a `packages/xyz` directory. Do not add entries to `pnpm-workspace.yaml`.
The four existing packages (`bff`, `common`, `fe`, `api-client`) are the complete set.

### 1.6 Never touch generated files

The following files are generated automatically and must not be edited by hand:

```
packages/bff/src/generated/routes.ts       ← generated by tsoa
packages/bff/src/generated/swagger.json    ← generated by tsoa
packages/api-client/src/generated/**       ← generated by openapi-generator
packages/api-client/dist/**                ← compiled output
packages/bff/dist/**                       ← compiled output
packages/common/dist/**                    ← compiled output
```

To regenerate routes after changing a controller: run `pnpm --filter bff tsoa`.
To regenerate the api-client after changing swagger.json: run `pnpm --filter api-client generate`.

### 1.7 No `console.log` in production code

Use `pino` (already installed). The `PrismaService` and server bootstrap already
import it. Inject or import the logger — never use `console.*` in `src/`.

`console.*` is only acceptable inside test files (`*.test.ts`, `*.spec.ts`).

## 2. TypeScript Rules

### 2.1 Strict mode is already on — keep it that way

`tsconfig.base.json` has `"strict": true`. Do not weaken it. Do not add
`"strictNullChecks": false` or `"noImplicitAny": false` to any tsconfig.

### 2.2 Return types must be explicit on all service and controller methods

```typescript
// BAD
async createCampaign(dto: CreateCampaignDto) {
  ...
}

// GOOD
async createCampaign(dto: CreateCampaignDto): Promise<CampaignDetailDto> {
  ...
}
```

Private helper methods inside a class may omit the return type only if the
type is trivially inferrable (e.g. `private buildWhereClause() { return { ... } }`).

### 2.3 Use `Decimal` from `@prisma/client` for all money values

Never use `number` for monetary amounts in service layer code. Prisma returns
`Decimal` objects for `NUMERIC` columns. Use `Decimal` arithmetic, not `Number()` coercion.

```typescript
// BAD — loses precision
const fee = Number(escrow.balance_usdc) * 0.05;

// GOOD
import { Decimal } from '@prisma/client/runtime/library';
const fee = escrow.balance_usdc
  .mul(campaignFeePercent)
  .div(100)
  .toDecimalPlaces(6, Decimal.ROUND_DOWN);
```

### 2.4 Enums must come from Prisma — not redefined

The Prisma schema defines all enums (`CampaignStatus`, `Currency`, `TransactionType`, etc.).
Import them from `@prisma/client`. Do not redefine equivalent string union types elsewhere.

```typescript
// BAD
type CampaignStatus = 'created' | 'funded' | 'samples_sent';

// GOOD
import { CampaignStatus } from '@prisma/client';
```

### 2.5 `unknown` in catch blocks — always

```typescript
// BAD
} catch (e) {
  logger.error(e.message);
}

// GOOD
} catch (error: unknown) {
  const message = error instanceof Error ? error.message : String(error);
  logger.error({ error: message }, 'Operation failed');
}
```

## 3. Architecture Rules

### 3.1 Layer boundaries are strict

```
Controller → Service → (Prisma | external service)
```

- **Controllers** handle HTTP: parse input, call one service method, return DTO. No business logic.
- **Services** contain all business logic. No direct HTTP concepts (`req`, `res`, status codes).
- **Workers/Jobs** call service methods. They do not contain business logic themselves.
- **Utils** are pure functions with no side effects and no DI dependencies.

Crossing layers (e.g., a service importing a controller, a worker importing another worker) is forbidden.

### 3.2 DTOs live in `packages/common`

All request/response DTO classes go in `packages/common/src/dtos/`.
The `bff` package imports them via the `common` workspace alias.
Do not define DTO classes inside `packages/bff/src/`.

**One file per domain:** `campaign.dto.ts`, `auth.dto.ts`, `wallet.dto.ts`, etc.
The files already exist — add to them, do not create duplicates.

### 3.3 One service per domain

Match the existing scaffolded service files:

```
admin.service.ts         campaign.service.ts      coa.service.ts
configuration.service.ts contribution.service.ts  email.service.ts
lab.service.ts           leaderboard.service.ts   notification.service.ts
ocr.service.ts           sample.service.ts        solana.service.ts
storage.service.ts       test.service.ts          user.service.ts
wallet.service.ts        auth.service.ts
```

Do not create new service files unless a domain is genuinely absent from this list.
If you need to share logic between two services, extract a pure utility function in
`packages/bff/src/utils/` — do not create a shared service just to avoid duplication.

### 3.4 DI registration goes in `container.ts` — nowhere else

All `tsyringe` registrations (`container.register`, `container.registerSingleton`)
belong in `packages/bff/src/container.ts`. Do not scatter registrations across files.

### 3.5 All DB operations that touch money use `SERIALIZABLE` isolation

Any Prisma transaction that debits or credits a `LedgerAccount`, `CampaignEscrow`,
or `FeeAccount` must use:

```typescript
await this.prisma.$transaction(
  async (tx) => {
    // ...
  },
  {
    isolationLevel: Prisma.TransactionIsolationLevel.Serializable,
  }
);
```

This is non-negotiable. The spec calls this out explicitly. A `READ COMMITTED`
transaction on a balance update is a bug, not a performance optimization.

### 3.6 LedgerTransaction rows are append-only

Never call `tx.ledgerTransaction.update()` or `tx.ledgerTransaction.delete()`.
The only exception is updating `status` from `pending` to `confirmed` or `failed`
on a withdrawal transaction — this is the one permitted update because the row's
existence was already recorded, only the outcome changes.

### 3.7 Config values are always read through `ConfigurationService`

Never hardcode values that exist in the `configuration` table
(`platform_fee_percent`, `min_contribution_usd`, `max_file_size_bytes`, etc.).
Always call `this.configService.get('key')` and type the result.

The `ConfigurationService` should cache values in memory with a short TTL (60s is fine).
Do not query the DB on every request for config.

## 4. Error Handling Rules

### 4.1 Use only the defined error classes

The spec defines exactly these error classes (section 6). Implement them in
`packages/bff/src/utils/errors.ts` and use only them:

```
ValidationError        → 400
AuthenticationError    → 401
AuthorizationError     → 403
NotFoundError          → 404
ConflictError          → 409
InsufficientBalanceError → 422
RateLimitError         → 429
InternalError          → 500
```

Do not `throw new Error('something went wrong')` anywhere in service code.
Do not create additional error classes. Map every failure to one of the above.

### 4.2 Error middleware is the only place that calls `res.status()`

Controllers never set HTTP status codes directly. They return a value.
tsoa + the error middleware handle status mapping.

### 4.3 Never swallow errors silently

```typescript
// FORBIDDEN
try {
  await riskyOperation();
} catch {
  // ignore
}

// REQUIRED — at minimum, log it
try {
  await riskyOperation();
} catch (error: unknown) {
  logger.error({ error }, 'riskyOperation failed — [explain consequence]');
  // then decide: rethrow, return fallback, or handle
}
```

The only legitimate reason to catch-and-not-rethrow is when the error is an
expected edge case that has a defined fallback (e.g., duplicate deposit signature
on the scanner idempotency guard).

## 5. Money and Transaction Safety Rules

These rules exist because bugs here are not "fix it in the next PR" problems —
they are real financial losses.

### 5.1 The balance check constraint is your last line of defense — not your first

The DB has `CHECK (balance >= 0)`. Do not rely on it as your primary guard.
Always check the balance in application code before attempting a debit, and
throw `InsufficientBalanceError` explicitly. The DB constraint is a safety net,
not the business rule.

### 5.2 Debit before credit — always

In any transaction that moves funds from A to B:

1. Debit A first
2. Verify the debit succeeded (check constraint will enforce, but read the result)
3. Credit B

If the transaction rolls back after the debit but before the credit, the funds
are safe because nothing left A. Never credit B first.

### 5.3 The `ProcessedDepositSignature` insert is the atomic gate

In the deposit scanner, the `ProcessedDepositSignature` insert must happen inside
the same DB transaction as the ledger credit. If the insert fails with a unique
constraint violation, treat it as "already processed" and skip silently (do not
throw, do not retry, do not credit). This is the idempotency guard.

### 5.4 Withdrawal worker: never retry a submitted on-chain transaction

If the on-chain transfer call returns a signature (even if confirmation times out),
the transaction may have landed. Before retrying, check the chain for the signature.
Do not blindly resubmit — this is how double-spends happen.

### 5.5 Fee calculation must use floor rounding

```typescript
// The spec formula — do not alter the rounding direction
const feeUsdc = escrow.balance_usdc
  .mul(campaign.platformFeePercent)
  .div(100)
  .toDecimalPlaces(6, Decimal.ROUND_DOWN); // floor, not round

const payoutUsdc = escrow.balance_usdc.sub(feeUsdc);
```

Rounding up on fees would silently over-charge creators. Round down always.

## 6. File and Naming Conventions

Follow the existing boilerplate patterns exactly. Do not introduce new conventions.

### 6.1 File naming

| Type               | Pattern                       | Example                      |
| ------------------ | ----------------------------- | ---------------------------- |
| Service            | `{domain}.service.ts`         | `campaign.service.ts`        |
| Controller         | `{domain}.controller.ts`      | `campaign.controller.ts`     |
| Worker             | `{domain}.worker.ts`          | `ocr.worker.ts`              |
| Job                | `{domain}.job.ts`             | `deadline-monitor.job.ts`    |
| DTO                | `{domain}.dto.ts` in `common` | `campaign.dto.ts`            |
| Util               | `{name}.util.ts`              | `jwt.util.ts`                |
| Test (unit)        | `{name}.service.test.ts`      | in `src/services/__tests__/` |
| Test (integration) | `{scenario}.test.ts`          | in `tests/integration/`      |

### 6.2 Class and method naming

- Service methods: camelCase verbs — `createCampaign`, `refundContributions`
- Controller methods: camelCase, matching the action — `contribute`, `shipSamples`
- DTO classes: PascalCase with suffix — `CreateCampaignDto`, `CampaignDetailDto`
- Error classes: PascalCase with suffix — `ConflictError`, `NotFoundError`

### 6.3 tsoa decorators must match the spec's API table exactly

When implementing a controller method, the path, HTTP verb, and auth requirement
must match section 9 of the spec exactly. Do not invent route variations.

```typescript
// Spec says: POST /campaigns/:id/lock  JWT (own campaign)
@Post('{id}/lock')
@Security('jwt')
async lockCampaign(@Path() id: string, @Request() req: ExpressRequest): Promise<CampaignDetailDto> {
```

## 7. Testing Rules

### 7.1 Unit tests: Vitest, integration tests: Jest

- Unit tests (`src/**/__tests__/*.test.ts`) use Vitest — already configured
- Integration tests (`tests/integration/*.test.ts`) use Jest + ts-jest
- Do not use Jest in unit test files or Vitest in integration test files

### 7.2 Mock only at the boundary

In unit tests, mock only external dependencies (Prisma, Redis, Solana RPC, S3, SMTP).
Do not mock other services that you own — test the interaction or write an integration test.

```typescript
// GOOD — mock the Prisma boundary
vi.mock('../prisma.service');

// BAD — mocking your own service to test another of your services
vi.mock('../campaign.service');
```

### 7.3 Integration tests must not share state

Each integration test file gets an isolated DB (truncate all tables in `beforeEach`
or use DB transactions that roll back). Never assume the DB is empty from a previous
test file's `afterAll`.

### 7.4 Test the spec's scenarios directly

The 10 integration test scenarios in section 13 of the spec are your test plan.
Name the test files after them:

```
tests/integration/scenario-01-happy-path.test.ts
tests/integration/scenario-02-refund-deadline.test.ts
...
tests/integration/scenario-10-concurrent-contributions.test.ts
```

Do not skip scenarios. If a scenario depends on on-chain behavior (scenarios 7, 9),
use the devnet test wallet specified in env vars.

## 8. Environment and Configuration Rules

### 8.1 All env var access goes through `env.config.ts`

`packages/bff/src/config/env.config.ts` uses `envalid` (already installed).
Every environment variable must be declared and validated there.
Do not call `process.env.WHATEVER` directly anywhere else in the codebase.

```typescript
// FORBIDDEN in service files
const rpcUrl = process.env.SOLANA_RPC_URL;

// REQUIRED
import { env } from '../config/env.config';
const rpcUrl = env.SOLANA_RPC_URL;
```

### 8.2 Secrets are never logged

The following values must never appear in any log output:

- `MASTER_WALLET_PRIVATE_KEY`
- `ENCRYPTION_KEY`
- `JWT_SECRET` / `JWT_REFRESH_SECRET`
- Any `encrypted_private_key` column value
- Any decrypted private key in memory

When logging errors from crypto or Solana operations, log the operation name
and error message only — not the key material involved.

### 8.3 The master wallet private key is for background jobs only

`env.MASTER_WALLET_PRIVATE_KEY` may only be accessed in:

- `packages/bff/src/workers/withdrawal.worker.ts`
- `packages/bff/src/services/solana.service.ts` (for the sweep operation)

It must not be imported, accessed, or passed as a parameter anywhere else.

## 9. When You Are Stuck or Uncertain

### 9.1 Spec gap → stop and ask

If you encounter a situation the spec does not cover and a conservative default
is not obvious, **stop and ask the user** before proceeding. Do not invent behavior
for financial flows, state transitions, or security decisions.

Write your question as:

```
SPEC GAP FOUND
Location: [section/flow you were implementing]
Gap: [what is not specified]
Options I see: [list 2-3 concrete options]
Blocking: [yes/no — can I implement anything else while waiting?]
```

### 9.2 Spec gap → conservative default (non-financial)

For non-financial, non-security gaps (e.g., a missing sort order, a response field
that's ambiguous), apply the most conservative/restrictive interpretation, implement
it, and leave a comment:

```typescript
// SPEC GAP: Section 9.3 does not specify sort order for COAs within a campaign.
// Conservative assumption: sort by uploaded_at ASC (oldest first).
// If this needs to change, update the orderBy clause here.
```

### 9.3 ESLint/TypeScript error you cannot resolve → stop and ask

Do not suppress. Do not work around it with a cast you cannot justify.
Describe the error and the code you were trying to write. Ask for guidance.

### 9.4 Compile before moving on

After completing each service file, run:

```
pnpm --filter bff build
```

Fix all TypeScript errors before moving to the next file.
Do not accumulate a backlog of type errors across multiple files.
